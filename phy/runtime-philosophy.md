# Soft Runtime Philosophy
Version: 1.0

> This document is the architectural contract for the Soft Runtime.
> The Runtime MUST follow this document. Whenever this document conflicts
> with an implementation, the implementation is wrong.

---

# Core Philosophy

Soft is NOT a JavaScript Framework.

Soft is a Programming Language that targets TypeScript.

The Compiler owns the application.

The Runtime executes compiler metadata.

The developer writes Business Logic only.

The compiler writes Infrastructure.

---

# Golden Rule

The Developer Writes Intent.

The Compiler Writes Implementation.

Example:

```soft
saveCustomer();
```

The compiler decides:

- Dependency registration
- DOM bindings
- Rendering metadata
- Bootstrap
- Routing
- Lifecycle
- Event wiring

The developer never writes those.

---

# Runtime Responsibilities

The Runtime is NOT a compiler.

The Runtime never parses templates.

The Runtime never parses expressions.

The Runtime never discovers dependencies.

The Runtime executes metadata generated by the compiler.

---

# Compiler Responsibilities

The compiler owns:

- Parsing
- Semantic Model
- Binder
- Dependency Graph
- Routing
- Bootstrap generation
- Component metadata
- Generated TypeScript

---

# Runtime Responsibilities

The Runtime owns:

- Object Proxies
- Lifecycle execution
- DOM updates
- Event dispatching
- Navigation
- Service resolution
- Error pipeline

Nothing else.

---

# Bootstrap

A Soft application never starts manually.

Forbidden:

```ts
new Todo();
render();
Soft.start();
```

Correct:

Developer executes

```
soft dev
```

Compiler performs:

1. Scan project
2. Discover @Page
3. Build Manifest
4. Generate bootstrap
5. Generate routes
6. Generate runtime metadata
7. Emit TypeScript
8. Invoke tsc

Browser receives generated application.

---

# Page Discovery

Every executable page MUST have

```soft
@Page("/")
```

Compiler scans every .s file.

Pages are automatically registered.

Developer never registers routes.

---

# Manifest

Compiler generates

ApplicationManifest

Contains:

- Pages
- Components
- Services
- Routes
- Metadata
- Dependency Graph

The Runtime reads Manifest only.

---

# Objects ARE State

Soft has NO Store.

Soft has NO Signals.

Soft has NO Observable objects.

The object itself is the state.

Example

```ts
class Customer{

    name="";
    age=0;

}
```

This object IS the state.

---

# Automatic Reactivity

Compiler wraps every state object.

Developer never calls:

reactive()

signal()

observable()

ref()

makeObservable()

Forbidden forever.

Generated code becomes

```ts
const customer =
SoftReactive.wrap(new Customer());
```

The developer never sees this.

---

# Proxy Engine

Every wrapped object is intercepted.

Setter

↓

Change Dispatcher

↓

onChange Pipeline

↓

Dependency Graph

↓

Rendering Engine

↓

DOM Patch

No manual notification.

---

# Arrays

Arrays are reactive.

Intercept:

push

pop

shift

unshift

splice

sort

reverse

length

Compiler knows affected bindings.

---

# Collections

Future:

Map

Set

WeakMap

WeakSet

Compiler metadata drives updates.

---

# Change Detection

Soft has NO Dirty Checking.

Soft has NO polling.

Soft has NO render loop.

Every change starts from a Proxy.

---

# onChange API

Developer may subscribe.

```ts
customer.onChange(
    x=>{

        x.stopUIRender();

    },
    p=>p.name,
    p=>p.age
);
```

Compiler generates strongly typed subscriptions.

Never string-based.

---

# Property API

Also valid

```ts
customer.name.onChange(...)
```

Compiler generates required wrappers.

---

# Dependency Graph

Compiler already knows

customer.name

↓

TextNode 15

customer.age

↓

Visible Attribute

No runtime discovery.

---

# Rendering

No Virtual DOM.

No Fiber.

No Tree Diff.

No Reconciliation.

Compiler already knows exactly which DOM node depends on which property.

Updating

customer.name

must update ONLY the affected text node.

Nothing else.

---

# Component Ownership

Component owns state.

Runtime owns component lifetime.

Developer never creates pages manually.

---

# Lifecycle

OnInit

OnBeforeRender

OnAfterRender

OnDestroy

Compiler wires lifecycle.

---

# Events

Compiler wires

@click

@change

@submit

Runtime dispatches only.

---

# Services

Constructor Injection only.

Compiler registers services.

Runtime resolves services.

No provider registration.

---

# Routing

Compiler discovers @Page.

Compiler builds routing table.

Runtime only navigates.

---

# Forms

Compiler creates metadata.

Runtime executes metadata.

---

# Error Pipeline

try/catch

↓

Method Error

↓

Class Error

↓

Global Error

↓

Crash

Developer never wires pipeline.

---

# DOM Ownership

Runtime owns DOM.

Developer never calls:

querySelector

appendChild

replaceChild

for framework-generated nodes.

---

# Performance Rules

No Virtual DOM.

No Runtime Reflection.

No Runtime Dependency Discovery.

No Template Parsing.

No Expression Parsing.

Batch DOM updates.

Reuse DOM references.

Avoid allocations.

---

# Runtime Size

Target

<30KB before compression.

---

# Soft vs React

React

Developer owns rendering.

Soft

Compiler owns rendering.

---

# Soft vs Angular

Angular

Runtime discovers changes.

Soft

Compiler discovers dependencies.

Runtime executes only.

---

# Mandatory Rules

The developer never starts the application.

The developer never creates pages manually.

The developer never registers routes.

The developer never notifies the framework about state changes.

The developer never calls render().

The compiler owns infrastructure.

The runtime owns execution.

Business code remains clean.

---

# Final Principle

Business First.

Compiler Second.

Runtime Third.

Everything else is implementation detail.
